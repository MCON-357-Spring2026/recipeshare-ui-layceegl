How does App Router work

Folder rule

app/page.js is the home page.

Layout rule

app/layout.js wraps every page.

Important rule

If you want useState or useEffect, the component must say "use client".

Server vs Client in the App Router, files are server components by default, which render on the server and are great for loading data securely. Client components run in the browser and are required when you use interactivity like useState, useEffect, or click handlers.

Very short JavaScript refresher

Objects & JSON

The way you define an object in JavaScript is JSON. JSON stands for JavaScript Object Notation — it was literally invented to describe JS objects. When your Flask API returns JSON, JavaScript can read it directly as a plain object. 

const recipe = {
  id: 1,
  title: "Soup"
};

// This is also valid JSON — same syntax!

Named Functions

function greet(name) {
  return `Hello ${name}`;
}

Arrow Functions (Lambdas)

Arrow functions are JavaScript's version of Java lambdas — a compact way to write a function without the function keyword. Arrow functions follow almost the same syntax as Java lambdas, but with a fat arrow => instead of -> . They are used everywhere in React: event handlers, .map() / .filter() callbacks, and useEffect bodies. 

// Traditional function
function add(a, b) {
  return a + b;
}

// Arrow function — more concise
const add = (a, b) => a + b;

// Single parameter — parentheses optional
const double = n => n * 2;

// Multi-line body needs curly braces + return
const greet = (name) => {
  const msg = `Hello, ${name}!`;
  return msg;
};

// Returning an object literal — wrap in ( )
const toObj = (title) => ({ title });

// Common in React callbacks
recipes.map(r => <RecipeCard key={r.id} recipe={r} />);

Collections: map, filter, reduce

You just learned Java Streams — map, filter, and reduce are the exact same idea in JavaScript. There is no separate "stream" step; you call these methods directly on any collection. Same mental model, slightly simpler syntax. 

const recipes = [
  { id: 1, title: "Soup",  calories: 200 },
  { id: 2, title: "Cake",  calories: 500 },
  { id: 3, title: "Salad", calories: 150 },
];  // array of objects

// map  → transform every element (like Stream.map)
const titles = recipes.map(r => r.title);
// ["Soup", "Cake", "Salad"] -> array of strings

// filter → keep only matching elements (like Stream.filter)
const light = recipes.filter(r => r.calories < 300);
// [{Soup}, {Salad}] -> array of objects, but only the light ones

// reduce → collapse to a single value (like Stream.reduce)
const total = recipes.reduce(
  (sum, r) => sum + r.calories, 0
);
// 850 -> a single number - total calories

// chain them, just like Java streams
const lightTitles = recipes
  .filter(r => r.calories < 300)
  .map(r => r.title);
// ["Soup", "Salad"]
Main message: React rendering a list is just .map() over an array of data — the same stream-style thinking you already know from Java, applied directly to arrays.

What is JSX?

JSX is the syntax React uses to describe what the UI should look like. It looks like HTML, but it lives inside a JavaScript file and is actually JavaScript. The key difference is that JSX lets you drop JavaScript expressions directly into your markup using curly braces { }.

JSX vs HTML

HTML is a static document format. JSX is compiled by React into JavaScript function calls, so you get the familiar look of HTML with the full power of JS behind it. A few things are slightly different: class becomes className, and every tag must be closed (e.g. <br />).

Embedding JavaScript with { }

Anything inside curly braces is evaluated as a JavaScript expression — a variable, a calculation, a function call, even a .map(). 

const title = "Chicken Soup";
const servings = 4;

export default function RecipeCard() {
  return (
    <div>
      <h2>{title}</h2>
      <p>Serves: {servings}</p>
      <p>Double batch: {servings * 2} servings</p>
    </div>
  );
}
Rule of thumb: plain text → just type it; dynamic value → wrap it in { }. Think of { } as an escape hatch from HTML-land back into JavaScript-land.

React Hooks

Hooks are special functions built into React that let your component do things — remember values, run code at the right time, share data, and more. They always start with the word use (e.g. useState, useEffect). You can only call hooks at the top level of a component, never inside loops or conditions. In this lesson we use two: useState to store data, and useEffect to fetch it (we'll introduce useEffect right before we need it).

useState

useState lets a component remember a value between renders. You give it an initial value and it hands you back two things: the current value, and a function to update it. Whenever you call the update function, React re-renders the component with the new value — this is what makes the page feel "live". Think of it like a variable that React is watching: change it through the setter and the UI automatically reflects the new state. Unlike a regular variable, state survives re-renders and always triggers the UI to update when it changes.

Syntax

// General pattern
const [value, setValue] = useState(initialValue);

// Example — a counter
const [count, setCount] = useState(0);

// Later, in a button handler:
setCount(count + 1);  // triggers a re-render

The [thing, setThing] convention

The naming convention [thing, setThing] is not enforced by React — it is just a very strong community convention. The first item is always the current value; the second is the updater function. 

// For a list of recipes:
const [recipes, setRecipes] = useState([]);

// For a loading flag:
const [loading, setLoading] = useState(true);

// For an error message:
const [error, setError] = useState("");
Rule: never modify state directly (e.g. recipes.push(...)). Always call the setter (setRecipes(...)) so React knows something changed and can re-render.

Step 3: Add state to RecipeList with useState

A hardcoded list isn't very useful. Let's use useState to store the recipes inside the component so they can change later. Remember — state is a variable that React watches: when it changes, the UI re-renders automatically.

Because we're using a hook (useState), we must declare this a client component with "use client" at the very top of the file. 

"use client";

import { useState } from "react";

export default function RecipeList() {
  const [recipes, setRecipes] = useState([
    { id: 1, title: "Chicken Soup" },
    { id: 2, title: "Caesar Salad" },
    { id: 3, title: "Chocolate Cake" },
  ]);

  return (
    <ul>
      {recipes.map((recipe) => (
        <li key={recipe.id}>{recipe.title}</li>
      ))}
    </ul>
  );
}

Three new things at once

  • "use client" — opts this component into the browser (required for hooks)
  • useState([...]) — stores the list; starts with hardcoded data for now
  • .map() in JSX — turns the array into <li> elements

The key prop

Each <li> has a key={recipe.id}. React uses this to efficiently update only the items that changed. Always give list items a unique key — React will warn you if you forget.

Check: the page should still show the same three recipes — but now they live in state inside RecipeList and are rendered with .map().

Introducing useEffect

useEffect lets you run code after the component renders — perfect for things that shouldn't block the UI from appearing, like fetching data from an API. You give it a function (the "effect") and an optional dependency array that controls when it runs. Passing an empty array [] means "run this once, right after the first render" — exactly what we want for an initial data load. Without the array, the effect would re-run after every single render, which is almost never what you want for a fetch call.

Shape of useEffect

useEffect(() => {
  // code that runs after render
}, [/* dependency array */]);

Dependency array cheat-sheet

  • [] — run once on mount (first render only)
  • [value] — re-run whenever value changes
  • omitted — re-run after every render
Common mistake: forgetting the [] on a fetch effect causes an infinite loop — fetch updates state → state change triggers a re-render → re-render triggers the effect again → repeat forever.

Common issue: CORS - Cross-Origin Resource Sharing

CORS is a browser security feature that controls how a web page can request resources from a different domain. The frontend runs on localhost:3000 and the Flask backend often runs on localhost:5000. Different ports count as different origins,so the browser blocks requests unless the backend explicitly allows it via CORS. When CORS is enabled on the backend, it sends special headers that tell the browser:

"I know this request is coming from a different origin, but it's okay, let it through."

This is an example HTTP response header that allows requests from our frontend:

Access-Control-Allow-Origin: http://localhost:3000

To enable CORS on the Flask side pull in this change into your local backend repo that should be using auth branch by typing: 

git pull upstream auth

Then, install the Flask-CORS package and add it to your Flask app 

pip install -r requirements.txt

In your Flask app in app/.__init__.py you will see this change:

from flask_cors import CORS

app = Flask(__name__)
CORS(app) # <---- is added to enable CORS for all routes, allowing the frontend to fetch data from the backend without CORS errors.

Step 4: Fetch real data from Flask inside RecipeList

Now replace the hardcoded initial state with an empty array, and add a useEffect to fetch the real data from the Flask API after the component first renders. 

"use client";

import { useState, useEffect } from "react";

export default function RecipeList() {
  const [recipes, setRecipes] = useState([]);

  useEffect(() => {
    fetch("http://localhost:5000/api/recipes")
      .then((response) => response.json())
      .then((data) => setRecipes(data))
      .catch((error) => console.log("Error fetching recipes:", error));
  }, []);

  return (
    <ul>
      {recipes.map((recipe) => (
        <li key={recipe.id}>{recipe.title || recipe.name}</li>
      ))}
    </ul>
  );
}
Notice the empty array: [] means this effect runs exactly once — right after RecipeList first appears on the page.

Step 5: Lift the fetch up to page.js (passing props)

Having the fetch inside RecipeList works, but pages often own the data and pass it down to components. This is a core React pattern called lifting state up. Move the fetch to page.js and pass the data to RecipeList as a prop.

Update app/page.js:

"use client";  //Add this to enable hooks in the page

import { useState, useEffect } from "react";
import RecipeList from "@/components/RecipeList";

export default function HomePage() {
  const [recipes, setRecipes] = useState([]);

  useEffect(() => {
    fetch("http://localhost:5000/api/recipes")
      .then((response) => response.json())
      .then((data) => setRecipes(data))
      .catch((error) => console.log("Error fetching recipes:", error));
  }, []);

  return (
    <main>
      <h1>My Recipes</h1>
      <p>Number of recipes: {recipes.length}</p>
      <RecipeList recipes={recipes} />
    </main>
  );
}

And simplify components/RecipeList.jsx — it no longer fetches, it just renders what it receives:

export default function RecipeList({ recipes }) {
  return (
    <ul>
      {recipes.map((recipe) => (
        <li key={recipe.id}>{recipe.title || recipe.name}</li>
      ))}
    </ul>
  );
}

What is a prop?

Props are arguments you pass into a component — just like method parameters in Java. The page owns the data and passes it down; the component just displays it. RecipeList doesn't care where the data came from, only that it receives an array.

Data flow

Flask API → page.js (state) → RecipeList (prop)
Now RecipeList no longer needs "use client" — it has no hooks. Only page.js needs it since that's where state and useEffect live.

Optional: add loading and error messages

This makes the UI feel more complete and also introduces realistic state handling.

"use client";

import { useEffect, useState } from "react";
import RecipeList from "@/components/RecipeList";

export default function HomePage() {
  const [recipes, setRecipes] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState("");

  useEffect(() => {
    fetch("http://localhost:5000/api/recipes")
      .then((response) => {
        if (!response.ok) {
          throw new Error("Failed to fetch recipes");
        }
        return response.json();
      })
      .then((data) => setRecipes(data))
      .catch((err) => setError(err.message))
      .finally(() => setLoading(false));
  }, []);

  return (
    <main>
      <h1>Recipes</h1>
      {loading && <p>Loading...</p>}
      {error && <p>Error: {error}</p>}
      {!loading && !error && <RecipeList recipes={recipes} />}
    </main>
  );
}

Upgrading the UI with Material UI (MUI)

Material UI (MUI) is the most popular React component library. Instead of writing raw HTML and CSS, you use pre-built, polished components like <Card>, <Typography>, and <Button> that follow Google's Material Design system. It saves a huge amount of CSS work and gives your app a professional look instantly.

Install MUI

Run this in your project folder:

npm install @mui/material @emotion/react @emotion/styled
MUI uses Emotion under the hood for styling — those two extra packages are required. You don't need to write any Emotion code yourself.

Step 6: Create a RecipeCard component

Instead of a plain <li>, each recipe will now be displayed in a MUI Card — a styled box with a title, a short description, and a button. Create components/RecipeCard.jsx: 

"use client";

import Card from "@mui/material/Card";
import CardContent from "@mui/material/CardContent";
import CardActions from "@mui/material/CardActions";
import Typography from "@mui/material/Typography";
import Button from "@mui/material/Button";

export default function RecipeCard({ recipe }) {
  return (
    <Card sx={{ marginBottom: 2 }}>
      <CardContent>
        <Typography variant="h6" component="h2">
          {recipe.title || recipe.name}
        </Typography>
        <Typography variant="body2" color="text.secondary">
          {recipe.description || "No description available."}
        </Typography>
      </CardContent>
      <CardActions>
        <Button size="small">View Recipe</Button>
      </CardActions>
    </Card>
  );
}

Key MUI components used

  • Card — the outer container box with a shadow
  • CardContent — padded area for the main content
  • CardActions — bottom area for buttons
  • Typography — text with built-in sizing and color variants
  • Button — a styled, accessible button

The sx prop

sx is MUI's shortcut for inline styles using a theme-aware syntax. sx={{ marginBottom: 2 }} means "apply 2 units of bottom margin" using MUI's spacing scale (1 unit = 8px by default, so this is 16px). Think of it as a smarter version of the HTML style attribute.

Step 7: Create a RecipeCardList component

Now add RecipeCardList.jsx component that uses MUI's layout components to arrange the cards in a responsive grid. Create components/RecipeCardList.jsx: 

import Grid from "@mui/material/Grid";
import RecipeCard from "./RecipeCard";

export default function RecipeCardList({ recipes }) {
  return (
    <Grid container spacing={3}>
      {recipes.map((recipe) => (
        <Grid size={{ xs: 12, sm: 6, md: 4 }} key={recipe.id}>
          <RecipeCard recipe={recipe} />
        </Grid>
      ))}
    </Grid>
  );
}

Grid layout explained

MUI Grid works like CSS Flexbox/Grid but with a 12-column system. xs={12} means "full width on mobile", sm={6} means "half width on tablet", md={4} means "one-third width on desktop". The cards will reflow automatically as the window resizes — for free.

Component tree

page.js
└── <RecipeCardList recipes={...} />
    └── <Grid container>
        └── <Grid item> × N
            └── <RecipeCard recipe={...} />
                └── <Card>, <Typography>, ...

Update page.js to use RecipeCardList

Swap RecipeList for RecipeCardList in app/page.js:

"use client";

import { useState, useEffect } from "react";
import Container from "@mui/material/Container";
import Typography from "@mui/material/Typography";
import RecipeCardList from "@/components/RecipeCardList";

export default function HomePage() {
  const [recipes, setRecipes] = useState([]);

  useEffect(() => {
    fetch("http://localhost:5000/api/recipes")
      .then((response) => response.json())
      .then((data) => setRecipes(data))
      .catch((error) => console.log("Error fetching recipes:", error));
  }, []);

  return (
    <Container maxWidth="lg" sx={{ mt: 4 }}>
      <Typography variant="h4" component="h1" gutterBottom>
        My Recipes
      </Typography>
      <RecipeCardList recipes={recipes} />
    </Container>
  );
}
Container is MUI's responsive page wrapper — it automatically centers the content and adds horizontal padding. maxWidth="lg" caps the width at 1200px. sx={{ mt: 4 }} adds top margin (4 × 8px = 32px).
