Why we need a proxy — CORS & session cookies

Our frontend runs on localhost:3000 and our Flask backend on localhost:5000. Even though both are on the same machine, the browser treats different ports as different origins. This triggers CORS (Cross-Origin Resource Sharing) restrictions.

Flask-Login uses session cookies to track who is logged in. For the browser to send a cookie to a different origin it needs credentials: "include" on every fetch, and the server must respond with:

Access-Control-Allow-Origin: http://localhost:3000   // exact origin, not *
Access-Control-Allow-Credentials: true

Configuring this correctly is error-prone. The simpler fix is a Next.js proxy rewrite:

❌ Without proxy

Browser → localhost:5000/api/…
Different origin → CORS check → cookie blocked → 302 redirect to login

✅ With proxy

Browser → localhost:3000/api/…
Same origin → no CORS → Next.js forwards to Flask → cookie works

How it works

Add a rewrites() function to next.config.mjs. Next.js intercepts any request matching the pattern and forwards it to Flask server-side:

// next.config.mjs
async rewrites() {
  return [
    { source: "/api/:path*",  destination: "http://localhost:5000/api/:path*" },
    { source: "/auth/:path*", destination: "http://localhost:5000/auth/:path*" },
  ];
}

Once the proxy is in place, all API calls use relative URLs — no port, no credentials: "include":

// lib/api.js
const BASE_URL = "";   // ← relative, routed through the proxy

const res = await fetch(`${BASE_URL}/api/recipes`, { ... });
⚠️ Restart required. Changes to next.config.mjs only take effect after restarting the dev server. After restarting you must also log out and log back in — the old session cookie was set directly on port 5000 and the proxy has never seen it.

In production — nginx takes over

The Next.js proxy is a development convenience. In production the Next.js app is built into static assets and served by a production-grade web server like nginx. Nginx plays the same proxy role — it sits in front of both the frontend and the backend and routes requests based on the URL path:

# nginx.conf (simplified)
server {
    listen 80;

    # Serve the Next.js frontend
    location / {
        proxy_pass http://nextjs:3000;
    }

    # Forward /api and /auth to Flask — same as next.config.mjs rewrites
    location /api/ {
        proxy_pass http://flask:5000;
    }
    location /auth/ {
        proxy_pass http://flask:5000;
    }
}

From the browser's perspective nothing changes — all requests still go to the same origin. The proxy concept is identical; only the tool doing the forwarding is different.

Key insight: the proxy pattern (same-origin forwarding to a backend) is universal. In development Next.js handles it. In production nginx (or a similar reverse proxy like Caddy or Apache) handles it. The frontend code — relative URLs, no credentials: "include" — stays exactly the same.

Protected routes

A protected route is a page that should only be accessible to logged-in users. In Next.js (App Router, client components) the pattern is a useEffect that runs on mount and redirects away if there is no auth:

"use client";

export default function CreateRecipePage() {
  const router = useRouter();

  useEffect(() => {
    if (!getAuth()) router.push("/login");  // ← redirect if not logged in
  }, []);

  // ... rest of the page
}
Page mounts → getAuth() === null? yes → router.push("/login") no → Render the page
This is UI-only protection. A determined user could still call your API directly (e.g. with curl or Postman). The server must also enforce authentication — Flask-Login's @login_required decorator handles this on the backend.

What getAuth() returns

It reads from localStorage. When logged in it returns an object; when logged out it returns null:

// logged in
getAuth() → { username: "alice", password: "…", userId: 6 }

// logged out
getAuth() → null

Dynamic routes & params in Next.js 15

A folder named [id] in the app/ directory creates a dynamic route segment. Whatever is in the URL at that position is passed to the page component as params:

app/
  recipes/
    [id]/
      page.js        →  /recipes/42
      edit/
        page.js      →  /recipes/42/edit

Why is params async in Next.js 15?

In older versions of Next.js, route parameters were handed to the page component as a plain synchronous object — your page rendered, params.id was already there, done. Next.js 15 changed this deliberately to support streaming server-side rendering (SSR).

In streaming SSR, the server starts sending HTML to the browser before it has finished computing everything. Parts of the page can arrive in chunks. Route parameters may belong to a chunk that hasn't been resolved yet when React first tries to render the component. To model this correctly, Next.js 15 made params a Promise — a container that says "the value will be here eventually, not necessarily right now."

Next.js 14 and earlier

params is a plain object. The page only renders after all route data is resolved. Simple, but the browser waits for everything before seeing anything.

Next.js 15+

params is a Promise. The page can start streaming to the browser immediately. Route data resolves asynchronously as it becomes available.

Because params is now a Promise, reading params.id directly gives you undefined — you're reading a property off a Promise object, not off the resolved value. You must unwrap the Promise first. React 19 provides use() for exactly this purpose inside Client Components:

// params is a Promise that resolves to e.g. { id: "42" }
// use() suspends the component until the Promise resolves,
// then returns the resolved value — here { id: "42" }
const { id } = use(params);
Why not just await params? You can't use await at the top level of a React component — hooks and component bodies must be synchronous. use() is the React-aware way to handle a Promise inside a component: it integrates with React's Suspense system so the component waits for the value without blocking the entire page.

⚠️ Next.js 15 gotcha — side-by-side comparison

❌ Broken — params.id is undefined
export default function EditPage({ params }) {
  const id = params.id;  // undefined in Next.js 15!
}
✅ Correct — unwrap with use()
import { use } from "react";

export default function EditPage({ params }) {
  const { id } = use(params);  // correctly resolved
}
Summary: params is a Promise because Next.js 15 uses streaming SSR — the server sends HTML in chunks before all data is resolved. React.use(params) unwraps the Promise inside a Client Component without breaking React's rendering rules. Forgetting use() is one of the most common Next.js 15 bugs — the page renders but id is silently undefined, causing fetches to fail with no obvious error.

Identifying ownership — comparing IDs, not usernames

The backend recipe payload contains user_id — the numeric primary key of the user who created it. There is no username string on the recipe object:

// Recipe payload from Flask
{ "id": 7, "title": "…", "user_id": 6, "prep_time": 30, … }

The login response returns the user's numeric id:

// Login response
{ "id": 6, "username": "alice", "email": "alice@example.com" }

We store that id as userId when saving auth to localStorage:

const data = await login(username, password);
saveAuth(username, password, data.id);  // store the numeric id

Then ownership is a simple integer comparison:

const isOwner = auth && auth.userId === recipe.user_id;
Common mistake: comparing auth.username === recipe.username always returns false because recipe.username is undefined — the field does not exist in the payload. Always compare the numeric IDs.

Using isOwner to conditionally render UI

// Only owners see Edit and Delete
{isOwner && (
  <>
    <Button onClick={() => router.push(`/recipes/${recipe.id}/edit`)}>Edit</Button>
    <Button color="error" onClick={handleDelete}>Delete</Button>
  </>
)}

Controlled forms & type coercion

In React, a controlled input is one whose value is driven by component state. Every keypress updates state, and state drives what the input displays. This gives you full control over the form data at all times.

const [formData, setFormData] = useState({ title: "", prep_time: "" });

function handleChange(e) {
  const { name, value } = e.target;
  setFormData((prev) => ({ ...prev, [name]: value }));
}

<TextField name="title" value={formData.title} onChange={handleChange} />
<TextField name="prep_time" value={formData.prep_time} onChange={handleChange} />

⚠️ HTML inputs always return strings

Even type="number" inputs return a string value. If your backend expects an integer, you must cast it before sending:

❌ Sends "30" (string) → server rejects
onSave(formData)
✅ Sends 30 (integer) → server accepts
onSave({ ...formData, prep_time: Number(formData.prep_time) })

One form, two purposes

The same RecipeForm component handles both create and edit. It receives an optional initialData prop — if provided, the form pre-fills; if not, it starts empty. The parent page decides what API call to make:

// Create page
<RecipeForm onSave={(data) => createRecipe(data)} />

// Edit page — form pre-fills from existing recipe
<RecipeForm initialData={recipe} onSave={(data) => updateRecipe(id, data)} />
Separation of concerns: the form manages field state and calls onSave. It doesn't know or care whether it is creating or editing — that logic lives in the page.

Parent-owned state & the async callback pattern

In React, state lives in the closest common ancestor that needs it. The home page owns the list of recipes — it fetched them and it renders them. Individual RecipeCard components display one recipe each, and a card needs to be able to trigger a deletion. But the card doesn't own the list, so it can't remove itself. The solution is a callback prop: the parent passes down a function that the child can call when something happens.

Why is deleting a recipe an async operation?

Deleting involves talking to a server. Network operations take time — milliseconds to seconds — and JavaScript cannot pause and wait for them without blocking the entire browser tab (freezing the UI). The solution is async/await:

  • async marks a function as one that will do work over time.
  • await says "pause this function here until the network call finishes, but let the rest of the browser keep running normally."
  • Only after the server confirms the delete (success response) do we update the UI.
Order matters. You must await the server call before updating state. If you update the UI first and the server call later fails, the item disappears from the screen but still exists in the database — the UI and the server are now out of sync.

The two-step delete

// RecipeCard.jsx
async function handleDelete() {
  // Step 1 — await the server. The function pauses here until Flask responds.
  // If the server returns an error, an exception is thrown and Step 2 never runs.
  await deleteRecipe(recipe.id);

  // Step 2 — server confirmed success. NOW we update the UI.
  // onDeleted is a function passed down from the parent page.
  onDeleted(recipe.id);
}

How the callback connects child to parent

// app/page.js — parent owns the recipe list
const [recipes, setRecipes] = useState([]);

function handleDeleted(id) {
  // Filter out the deleted recipe — React re-renders with the new list
  setRecipes((prev) => prev.filter((r) => r.id !== id));
}

// Pass the function down as a prop
<RecipeCardList recipes={recipes} onDeleted={handleDeleted} />
// RecipeCardList.jsx — passes it further down to each card
<RecipeCard recipe={recipe} onDeleted={onDeleted} />
// RecipeCard.jsx — calls it after the server confirms
async function handleDelete() {
  await deleteRecipe(recipe.id);  // waits for server ✓
  onDeleted(recipe.id);           // tells parent to remove it from state
}

The full async flow, step by step

1. User clicks Delete → handleDelete() is called
2. await deleteRecipe(7) → Browser sends DELETE /api/recipes/7 to Next.js proxy
3. Proxy forwards → Flask receives request, checks session cookie, deletes from DB
4. Flask responds 200 OK → await resolves — handleDelete continues
5. onDeleted(7) is called → Parent's handleDeleted(7) runs
6. setRecipes(prev => prev.filter(…)) → React re-renders the list without recipe 7
7. Card disappears → No page reload — only the changed list re-renders
Why no page reload? React only re-renders the components whose state changed. Calling setRecipes() tells React "the list changed" — it diffs the old and new list and removes just the one card from the DOM. The rest of the page is untouched.

What happens if the server call fails?

If Flask returns an error (e.g. 403 Forbidden because you don't own the recipe), deleteRecipe() throws an exception. Because onDeleted() comes after the await, it never runs — the card stays on screen and the list remains accurate. Always wrap this in a try/catch to show the user the error:

async function handleDelete() {
  try {
    await deleteRecipe(recipe.id);  // may throw
    onDeleted(recipe.id);           // only runs on success
  } catch (err) {
    alert(err.message);  // or set an error state and show an Alert
  }
}

Responsive card layout with CSS auto-fill

Instead of defining explicit breakpoints (2 columns on tablet, 3 on desktop…), CSS auto-fill with minmax handles this automatically:

gridTemplateColumns: "repeat(auto-fill, minmax(280px, 1fr))"

This tells the browser: "create as many columns as fit, each at least 280 px wide, sharing any leftover space equally."

Viewport ~600 px

2 columns × 280 px = 560 px fits → 2 columns

Viewport ~900 px

3 columns × 280 px = 840 px fits → 3 columns

Viewport ~1200 px

4 columns × 280 px = 1120 px fits → 4 columns

⚠️ The container must have an explicit width

Without width: "100%" on the grid container, the browser has no reference width to divide up — all cards collapse into a single column:

❌ No width — single column
<Box sx={{ display: "grid",
  gridTemplateColumns: "repeat(auto-fill, minmax(280px, 1fr))",
  gap: 3 }}>
✅ With width — fills the viewport
<Box sx={{ display: "grid",
  gridTemplateColumns: "repeat(auto-fill, minmax(280px, 1fr))",
  gap: 3,
  width: "100%" }}>

Error handling in async operations

Every async function that calls the API can throw. Without a try/catch the error is silently swallowed — the user sees nothing happen and has no idea what went wrong.

❌ No error handling — silent failure
async function handleSave(formData) {
  await createRecipe(formData);
  router.push("/");
  // if createRecipe throws, nothing happens
}
✅ With try/catch — user sees the error
async function handleSave(formData) {
  setError("");
  try {
    await createRecipe(formData);
    router.push("/");
  } catch (err) {
    setError(err.message);  // shown in an <Alert>
  }
}

Making error messages useful

The API helper should provide a meaningful message rather than a raw status code:

if (!res.ok) {
  const data = await res.json().catch(() => ({}));
  throw new Error(data.error || `Server error ${res.status} – are you logged in?`);
}

The .catch(() => ({})) guards against the case where the error response body is not valid JSON (e.g. a Flask redirect HTML page).
